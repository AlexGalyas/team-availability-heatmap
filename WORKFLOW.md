# Harness Workflow — операційний мануал (Frontend)

Це твій спосіб працювати з Claude Code так, щоб великі задачі не розвалювались
через переповнення контексту. Суть harness engineering: робота розбита на фази,
між фазами контекст **скидається** (`/clear`), а стан передається через
**handoff-артефакт** (`STATE.md`), і кожна фаза закривається **gate'ом** (тести +
typecheck + lint + build зелені).

Промпт engineering і context engineering працюють усередині одного вікна.
Harness — це шар *над* вікнами. Цей kit і є той шар.

---

## Ключова ідея в одному реченні

> Один контекст = одна фаза. Між фазами зберігаєш стан у `STATE.md`, чистиш
> контекст, відновлюєшся з `STATE.md`. Жодна фаза не вважається завершеною,
> поки `gate` не зелений.

---

## Структура артефактів у репо

```
.claude/commands/        # slash-команди = твоя упряж
  spec.md   plan.md   tasks.md   next.md   handoff.md   resume.md   gate.md
docs/
  constitution/          # незмінні правила проєкту (mission, tech-stack, conventions)
  specs/
    _templates/          # шаблони spec/plan/tasks/STATE
    <feature-slug>/      # одна папка на фічу
      spec.md            # ЩО будуємо + acceptance criteria (виконувані гейти)
      plan.md            # ЯК: архітектура, файли, рішення
      tasks.md           # розбивка на дрібні задачі з чекбоксами
      STATE.md           # ★ handoff-артефакт: жива пам'ять між сесіями
scripts/
  gate.sh                # phase gate: typecheck → lint → test → build
CLAUDE.md                # постійна пам'ять проєкту (читається щосесії)
```

`STATE.md` — серце harness. Це не документ для архіву, це робочий журнал, який
агент перечитує на старті кожної сесії і дописує наприкінці.

---

## Життєвий цикл фічі (повний прохід)

### Фаза 0 — Constitution (один раз на проєкт)
Заповнюєш `docs/constitution/`. Це те, що ніколи не змінюється від фічі до фічі:
стек, структура feature-based, патерни Next.js/Redux/SCSS Modules, вимоги до тестів.
Це підтягується у `CLAUDE.md`, тож агент знає правила гри без нагадувань.

### Фаза 1 — Spec  (`/spec <опис фічі>`)
Агент ставить тобі уточнюючі питання (Q&A), потім пише `spec.md`:
що будуємо, навіщо, acceptance criteria. Критерії формулюй так, щоб їх можна
було перевірити автоматично — це майбутні гейти, а не побажання.
**В кінці фази:** `/handoff` → `/clear`.

### Фаза 2 — Plan  (`/plan <feature-slug>`)
Свіжий контекст. `/resume` підтягує STATE. Агент читає `spec.md`, проектує
архітектуру: які роути/сторінки, дерево компонентів (Server vs Client),
фетчинг даних, UI-стани, стилі. Пише `plan.md`. Жодного коду ще.
**В кінці:** `/handoff` → `/clear`.

### Фаза 3 — Tasks  (`/tasks <feature-slug>`)
Агент б'є план на дрібні задачі (кожна ≈ один логічний коміт, проходить gate
самостійно). Пише `tasks.md` з чекбоксами.
**В кінці:** `/handoff` → `/clear`.

### Фаза 4 — Implement  (цикл `/next` ... `/next`)
Тут основна робота. Кожен `/next`:
1. читає `STATE.md` + `tasks.md`, бере наступну невиконану задачу,
2. імплементує **тільки її**,
3. ганяє `gate`,
4. якщо зелено — ставить чекбокс, оновлює `STATE.md`,
5. якщо контекст роздувся або фаза логічно завершена — `/handoff` → `/clear` → `/resume`.

Не намагайся зробити всю фічу в одному вікні. Роби 1–3 задачі, потім reset.
Це і є дисципліна, яка тримає якість.

### Фаза 5 — Ship
Коли всі чекбокси закриті і `gate.sh` зелений: PR → review → squash-merge → tag.
(Твій звичний фінал, тепер з гарантією, що кожен крок пройшов гейт.)

---

## Коли робити context reset (`/clear`)

Скидай контекст, коли спрацьовує будь-що з цього:
- завершилась логічна фаза (spec / plan / tasks / блок задач);
- зробив 2–3 задачі поспіль;
- агент почав «забувати» рішення або плутати файли (ранній симптом context rot);
- довгий debug-діалог засмітив вікно сирими логами/стектрейсами.

Завжди: `/handoff` (зберегти стан) → `/clear` → `/resume` (відновити стан).
`STATE.md` — єдине джерело правди, не людська пам'ять і не контекст вікна.

---

## Паралельна робота: git worktrees

Для незалежних фіч паралельно — окремий worktree на фічу, окрема Claude Code сесія:

```bash
git worktree add ../proj-feature-x -b feature/x
cd ../proj-feature-x && claude
```

Кожна сесія має власний контекст і власний `STATE.md` у своїй гілці. Нуль конфліктів
контексту між фічами. Це масштабує harness горизонтально.

---

## Subagents для чистоти контексту

Брудну розвідку («знайди, де визначено цей тип», «прочитай усі ці 8 файлів»)
делегуй sub-agent'у. Він повертає підсумок у головне вікно, а сирі дані лишаються
в його ізольованому контексті. Головне вікно лишається чистим для рішень.

---

## Дві залізні дисципліни

1. **Gate перед просуванням.** Жодна задача не «зроблена» без зеленого `gate.sh`.
   AI-код статистично містить вразливості частіше, ніж здається — гейт це твій
   запобіжник, а не формальність.
2. **STATE перед reset.** Ніколи не роби `/clear`, не зробивши `/handoff`.
   Втратиш стан — втратиш зчеплення між сесіями, і harness розвалиться.
