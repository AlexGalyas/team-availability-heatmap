# Spec: Team Availability Heatmap  (frontend)

slug: `team-availability-heatmap`
status: draft
owner: galyasolexandr

## Problem / Why
Команді важко швидко знайти спільне вікно часу, коли вільні всі (або більшість).
Опитування в чаті й ручне зведення таблиць — повільно й помилково. Потрібна
візуальна сітка «тиждень × години», де кожен фарбує свої вільні слоти, а зверху
накладається теплова мапа перетину — щоб за один погляд бачити «коли всі вільні».
Повторювані правила (напр. «щовівторка 9:00–12:00») прибирають рутину масового
заповнення.

## Scope
### In scope
- Сітка **Пн–Пт × 08:00–20:00, крок 60 хв** (5 днів × 12 годин = 60 слотів).
- **Drag-фарбування** свого шару доступності: затиснув і потягнув — виділив
  суцільний діапазон слотів; режим визначається першою клітинкою (фарбуємо/стираємо).
- **Теплова мапа перетину**: насиченість клітинки ∝ кількості вільних учасників у
  слоті; легенда інтенсивності; підсвічування слотів, де вільні всі.
- **Перемикач «хто я»** зі списку команди (без auth) — змінює редагований шар.
- **Форма «повторюване правило»** (Formik + Yup): день тижня + початок + кінець →
  масово заповнює мій шар у сітці.
- Чотири стани кожної поверхні: loading / empty / error / success.
- Збереження мого шару через mock route handlers (in-memory) + дзеркало в localStorage.

### Out of scope
- Реальна автентифікація / сесії (вибір учасника — простий dropdown).
- Реальна БД / зовнішній бекенд (тільки in-memory route handlers).
- Real-time мульти-юзер синхронізація (WebSocket/SSE) — дані інших учасників
  читаються разово зі сіду, без живих апдейтів.
- Повна DST-коректність TZ (див. Notes — використовуємо фіксований offset).
- Експорт/шеринг, інтеграції з календарями, мобільні жести понад базовий tap/drag.
- Слоти поза вікном 08:00–20:00 та вихідні (Сб/Нд).

## User flows & screens
Єдиний екран `/availability`:
1. Користувач відкриває сторінку → бачить сітку Пн–Пт × 08–20 з уже накладеною
   тепловою мапою (сід інших учасників) + легенду + перемикач учасника + форму правила.
2. Обирає себе у дропдауні «Я — …». Його збережений шар підсвічується поверх мапи.
3. **Малює доступність**: натискає на слот і тягне — виділяє діапазон. Перша клітинка
   задає напрям: якщо вона була порожня → фарбуємо, якщо зайнята → стираємо. Відпускає —
   зміни комітяться (mutation), теплова мапа перераховується.
4. **Повторюване правило**: у формі обирає день/початок/кінець (напр. Вт 09:00–12:00),
   тисне «Застосувати» → відповідні слоти масово додаються до його шару; мапа оновлюється.
5. Скидання мого шару (кнопка «Очистити мій вибір») — опційно, нескладно.

## UI states  (для кожної ключової поверхні)
### Heatmap grid
- **Loading**: skeleton-сітка тих самих розмірів (без layout shift), aria-busy.
- **Empty**: учасників немає / у всіх порожньо → сітка «холодна», підказка «Ще ніхто
  не позначив доступність».
- **Error**: запит доступності впав (5xx) → повідомлення + кнопка retry.
- **Success**: інтерактивна сітка з тепловою мапою та моїм шаром.

### Recurring-rule form
- **Loading**: submit вимкнено під час застосування (pending).
- **Empty/initial**: поля з дефолтами, submit активний лише при валідній формі.
- **Error**: помилки валідації на рівні полів (Yup) + помилка mutation (банер).
- **Success**: тост/повідомлення «Правило застосовано», сітка оновлена.

## Data
- **Server-rendered (Server Components + route handlers)**: початковий список
  учасників і їхні збережені слоти для першого рендеру сітки (без client-flash).
  Сід тримається in-memory на сервері route handler'ів.
- **Client (RTK Query)**:
  - `getMembers` — список учасників (id, name).
  - `getAvailability` — слоти всіх учасників (для теплової мапи).
  - `putMyAvailability` — зберегти/замінити мій шар (drag-commit та clear).
  - `applyRecurringRule` — масово додати слоти за правилом до мого шару.
  - Тег-інвалідація: мутації інвалідують `Availability` → мапа перераховується.
- **Global state (Redux Toolkit)**:
  - `currentMemberId` (хто я) — перемикач, дзеркалиться в localStorage.
  - Тимчасовий drag-стан (активний діапазон, режим paint/erase) під час малювання —
    локально, комітиться у мутацію на mouseup.

## Forms
- **Recurring rule** (Formik + Yup):
  - Поля: `weekday` (Mon–Fri), `startHour` (08–19), `endHour` (09–20).
  - Yup: всі обовʼязкові; `endHour > startHour`; межі в [08, 20]; вирівнювання на годину.
  - Помилки — на рівні полів; submit заблоковано, поки форма невалідна.

## Acceptance criteria  (майбутні гейти — перевірюваний вигляд)
- [ ] Сітка рендериться Пн–Пт × 08:00–20:00 (60 слотів), shell зі Server Component,
      без layout shift при гідрації.
- [ ] Drag по сітці виділяє суцільний діапазон; режим (paint/erase) задається першою
      клітинкою; mouseup комітить мій шар через RTK Query mutation.
- [ ] Теплова мапа: насиченість клітинки відповідає кількості вільних учасників;
      присутня легенда; слоти «вільні всі» візуально виділені.
- [ ] Перемикач учасника змінює редагований шар; вибір зберігається в localStorage
      і відновлюється після перезавантаження.
- [ ] Formik+Yup форма блокує невалідний submit, показує помилки полів; «Застосувати»
      масово заповнює мій шар згідно з правилом, мапа оновлюється.
- [ ] UTC-зберігання / локальне відображення задокументовано й узгоджено (фіксований
      offset, коректний перенос слота через межу доби).
- [ ] Кожна поверхня (grid, form) обробляє loading/empty/error/success.
- [ ] a11y: сіткою можна керувати з клавіатури (фокус по клітинках, Enter/Space —
      toggle, Shift+стрілки — діапазон); інпути підписані; видимий focus.
- [ ] Покриті тестами: hook підрахунку інтенсивності, hook drag-діапазону, чиста
      функція розгортання повторюваного правила в слоти (Vitest+RTL); критичний flow
      (намалював → застосував правило → бачу оновлену мапу) — Playwright.
- [ ] `bash scripts/gate.sh` зелений.

## Open questions
- **DST / межа доби при UTC↔локаль для повторюваного тижня.** Пропоную для MVP:
  фіксований offset з `Intl`/`getTimezoneOffset` без урахування переходів DST;
  слот зберігається як (день-тижня, година) в UTC, на дисплеї конвертується одним
  offset'ом, з коректним переносом через північ (слот може «виїхати» на сусідній день).
  → **Підтвердити, що фіксований offset прийнятний для MVP.**
- Чи потрібна кнопка «Очистити мій вибір» у MVP? (закладаю як nice-to-have).
- Скільки сід-учасників і які імена? (закладаю 5–6 фейкових).

## Notes / constraints
- **TZ**: канонічне сховище — UTC (день-тижня + година UTC). Відображення —
  у локальному поясі переглядача. DST ігноруємо (фіксований offset) — це MVP-спрощення,
  винесене в Out of scope/Open questions, щоб уникнути hydration mismatch і складності.
- **Hydration**: будь-яке форматування часу, залежне від локалі/offset переглядача,
  робити клієнтсько (узгодити SSR/CSR), щоб не ловити mismatch.
- **Перформанс**: 60 клітинок — дешево; drag не має ре-рендерити всю сітку щокадру
  (мемоізація клітинок, локальний drag-стан поза глобальним store до commit).
- **Проєкт ще не заскафолджено**: немає `package.json`/`src`. Перша задача implement-фази —
  підняти Next.js (App Router, TS strict) + інструменти (Redux/RTK Query, Formik/Yup,
  Vitest/RTL, Playwright, SCSS Modules) і `pnpm`-скрипти під `gate.sh`.
- **Responsive**: десктоп-first (сітка + бічна форма); на вузьких екранах форма під сіткою.
